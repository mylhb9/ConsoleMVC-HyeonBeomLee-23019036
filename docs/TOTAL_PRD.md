## 1. 배경 및 문제 정의

### 1.1 현재 상황 (As-Is)

가상의 반도체 회사 **S-Semi**는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등의 고객에게 납품하고 있다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.

현재는 엑셀과 메모장으로 주문을 관리하고 있어 다음 문제가 발생한다:
- 주문 처리 여부 파악 불가
- 공정 예약 완성 시점 불투명
- 재고가 충분함에도 불필요한 생산 공정 실행
- 주문량 급증으로 인한 관리 오류 빈발

### 1.2 핵심 문제 (Problem Statement)

> 엑셀·메모장 기반의 수기 관리로 인해 재고 현황, 생산 공정 상태, 주문 처리 결과를 실시간으로 파악할 수 없어 생산 효율이 저하되고 납품 지연이 발생한다.

### 1.3 해결하지 않을 경우 비용 (Cost of Inaction)

- 주문 누락 또는 중복 처리로 인한 고객 신뢰 하락
- 불필요한 생산 공정 실행으로 인한 원가 낭비
- 재고 현황 파악 실패로 인한 납기 지연

## 2. 목표 및 성공 지표

### 2.1 비즈니스 목표

- 콘솔 기반의 반도체 시료 생산주문관리 시스템 구축
- 주문 접수부터 출고까지 전 과정의 디지털 추적 체계 확립
- 재고 및 생산 현황 실시간 모니터링 가능화

### 2.2 사용자 목표

| 역할 | 목표 |
|------|------|
| 주문 담당자 | 고객 요청을 정확하게 주문서로 전환, 상태 추적 |
| 생산 담당자 | 시료 등록 및 주문 승인/거절 처리, 생산 큐 관리 |

### 2.3 KPI / 성공 지표

- 주문 상태 조회 응답 시간: **1초 이내**
- 시료 등록~출고 전 과정 콘솔 내 처리 완결률: **100%**
- 주문 상태 정합성 오류율: **0%** (상태 전이 규칙 준수)
- PoC 4개 항목 전체 완료율: **100%**

### 2.4 MVP 범위

**In-Scope:**
- 시료 관리 (등록/조회/검색)
- 시료 주문 (예약/승인/거절)
- 모니터링 (상태별 주문량, 재고 현황)
- 생산라인 조회 (FIFO 스케줄링)
- 출고 처리
- 데이터 영속성 (애플리케이션 재시작 후 데이터 유지)

**Out-of-Scope:**
- 웹/모바일 UI
- 고객 직접 접속 포털
- 이메일 자동 발송 기능
- 결제 처리

## 3. 사용자 정의

### 3.1 타겟 사용자 세그먼트

| 세그먼트 | 설명 |
|---------|------|
| 주문 담당자 | 고객 요청 접수, 주문서 작성 및 전달 |
| 생산 담당자 | 시료 등록, 주문 승인/거절, 생산 큐 관리 |

### 3.2 역할별 접근 권한

| 기능 | 주문 담당자 | 생산 담당자 |
|------|:----------:|:----------:|
| 시료 관리 (등록/조회/검색) | ○ | ○ |
| 시료 주문 (예약) | ○ | - |
| 주문 승인/거절 | - | ○ |
| 모니터링 | ○ | ○ |
| 생산라인 조회 | - | ○ |
| 출고 처리 | ○ | - |

### 3.3 사용자 여정

```
고객 시료 요청 (이메일)
    ↓
주문 담당자: 시료 주문 등록 (RESERVED)
    ↓
생산 담당자: 주문 승인/거절
    ├─ 거절 → REJECTED (종료)
    └─ 승인 → 재고 확인
                ├─ 재고 충분 → CONFIRMED (출고 대기)
                └─ 재고 부족 → PRODUCING (생산 중)
                                    ↓ (생산 완료)
                                CONFIRMED (출고 대기)
                                    ↓
주문 담당자: 출고 처리 → RELEASE (출고 완료)
```

## 4. 프로젝트 구성

본 과제는 **PoC 4개 + 본 프로젝트 1개**, 총 5개 Repository로 구성된다.

| # | PoC 항목 | Repository | 설명 |
|---|---------|-----------|------|
| 1 | MVC 스켈레톤 코드 | `ConsoleMVC` | Model/Controller/View 패키지 구조와 역할 분리 완성 |
| 2 | 데이터 영속성 처리 | `DataPersistence` | 선택 방식(파일/JSON/DB)으로 데이터 저장·불러오기 구조 구현 (CRUD 포함) |
| 3 | 데이터 모니터링 Tool | `DataMonitor` | 현재 저장된 데이터 상태를 콘솔에서 실시간 조회할 수 있는 관리자 도구 |
| 4 | Dummy 데이터 생성 Tool | `DummyDataGenerator` | Test용 Dummy Data 생성 도구 (연결된 DB에 추가) |
| 5 | 반도체 시료 생산주문관리 시스템 | `SampleOrderSystem` | 전체 기능 명세를 충족하는 최종 시스템 |

> 각 PoC PRD는 해당 프로젝트의 `docs/PRD.md` 참조

## 5. 시스템 아키텍처 개요

### 5.1 기술 스택

| 구분 | 기술 |
|------|------|
| 언어 | Java |
| 빌드 도구 | Gradle |
| 아키텍처 | Layered Architecture (MVC 기반) |
| UI | Console (CLI) |
| 데이터 저장 | 파일 / JSON / Oracle DB (DataPersistence PoC에서 결정) |
| 테스트 | JUnit 5 |

### 5.2 레이어 구조

```
Console Input
    ↓
[View Layer]        화면 출력 및 사용자 입력 처리
    ↓
[Controller Layer]  비즈니스 로직 흐름 제어
    ↓
[Service Layer]     도메인 로직 수행
    ↓
[Repository Layer]  데이터 저장소 접근 (CRUD)
    ↓
[Data Store]        파일 / JSON / DB
```

### 5.3 핵심 엔티티

```
Sample (시료)
  - sampleId   : String  (e.g. S-001)
  - name       : String  (e.g. 실리콘 웨이퍼-8인치)
  - avgProdTime: double  (평균 생산시간, min/ea)
  - yield      : double  (수율, 0.0 ~ 1.0)
  - stock      : int     (현재 재고 수량, ea)

Order (주문)
  - orderId    : String  (e.g. ORD-20260416-0043)
  - sampleId   : String
  - customerName: String
  - quantity   : int
  - status     : OrderStatus (RESERVED/REJECTED/PRODUCING/CONFIRMED/RELEASE)
  - createdAt  : LocalDateTime

ProductionQueue (생산 큐)
  - orderId    : String
  - sampleId   : String
  - deficitQty : int     (부족분)
  - actualProdQty: int   (실 생산량 = ceil(부족분 / (수율 * 0.9)))
  - estimatedTime: int   (총 생산 시간 = 평균 생산시간 * 실 생산량)
```

## 6. 주문 상태 흐름

```
[RESERVED] ──승인(재고 충분)──→ [CONFIRMED] ──출고 처리──→ [RELEASE]
     │
     ├──승인(재고 부족)──→ [PRODUCING] ──생산 완료──→ [CONFIRMED]
     │
     └──거절──→ [REJECTED]
```

| 상태 | 의미 |
|------|------|
| `RESERVED` | 주문 접수 완료 (출고 대기 상태) |
| `REJECTED` | 주문 거절 (정상 흐름 외, 모니터링 제외) |
| `PRODUCING` | 주문 승인 완료, 재고 부족으로 생산 중 |
| `CONFIRMED` | 주문 승인 완료, 재고 충분하여 출고 대기 중 |
| `RELEASE` | 출고 완료 |

## 7. 전체 기능 요구사항

### 7.1 시료 관리

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| 시료 등록 | 시료 ID, 이름, 평균 생산시간, 수율 입력하여 등록 | Must Have |
| 시료 조회 | 등록된 모든 시료 목록 + 현재 재고 수량 표시 | Must Have |
| 시료 검색 | 이름 등 속성으로 특정 시료 검색 | Must Have |

### 7.2 시료 주문

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| 시료 예약 | 시료 ID, 고객명, 주문 수량 입력 → RESERVED 상태로 등록 | Must Have |
| 주문번호 자동 생성 | `ORD-YYYYMMDD-NNNN` 형식으로 자동 부여 | Must Have |

### 7.3 주문 승인/거절

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| RESERVED 목록 조회 | 승인 대기 중인 주문 목록 표시 | Must Have |
| 주문 승인 | 재고 충분 → CONFIRMED, 재고 부족 → PRODUCING + 생산 큐 등록 | Must Have |
| 주문 거절 | 즉시 REJECTED 상태로 전환 | Must Have |

### 7.4 모니터링

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| 상태별 주문량 확인 | RESERVED/CONFIRMED/PRODUCING/RELEASE 건수 표시 | Must Have |
| 재고량 확인 | 시료별 재고 수량, 상태 표시 (여유/부족/고갈) | Must Have |

### 7.5 생산라인 조회

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| 현재 생산 현황 | 현재 처리 중인 주문 정보 표시 | Must Have |
| 대기 주문 확인 | FIFO 방식 생산 대기 큐 목록 표시 | Must Have |
| 실 생산량 계산 | `ceil(부족분 / (수율 * 0.9))` | Must Have |

### 7.6 출고 처리

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| 출고 가능 주문 조회 | CONFIRMED 상태 주문 목록 표시 | Must Have |
| 출고 실행 | 특정 주문에 대해 출고 처리 → RELEASE 상태로 전환 | Must Have |

## 8. 비기능 요구사항

### 8.1 성능

- 모든 콘솔 명령 응답 시간: **1초 이내**
- 생산라인 조회 시 실시간 진행률 표시 가능

### 8.2 데이터 영속성

- 애플리케이션 재시작 후에도 모든 데이터(시료, 주문, 생산 큐) 유지
- 데이터 저장 방식: 파일 / JSON / Oracle DB 중 선택 (DataPersistence PoC 결과 기반)

### 8.3 안전성

- 잘못된 입력값에 대한 예외 처리 및 재입력 유도
- 주문 상태 전이는 정의된 흐름 외 전환 불가

### 8.4 코드 품질

- 레이어드 아키텍처 준수 (Controller → Service → Repository)
- 단일 책임 원칙 준수
- 테스트 코드 작성 (단위 테스트 필수)

## 12. 용어 정의

| 용어 | 설명 |
|------|------|
| 시료 (Sample) | 반도체 공정을 통해 생산되는 제품 단위 |
